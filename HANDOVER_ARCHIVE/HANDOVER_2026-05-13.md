#### WF9 - Pickliste verarbeiten (ID: UtXagT58XYNwxUM5)

**Problem vorher:** Beim manuellen Auffüllen des Automaten musste WF7 für jedes Produkt
einzeln per Webhook aufgerufen werden.

**Lösung:** Neuer Workflow WF9. Der User erzeugt in der App eine Pickliste (PDF),
lädt sie in Google Drive Ordner „pickliste unbearbeitet" hoch → WF9 triggert automatisch
und verarbeitet alle Produkte auf einmal.

**Trigger (zwei Einstiegspunkte):**
- **Google Drive Trigger** (auto): Polling alle 60s, feuert bei neuer Datei in Ordner `1Djrp-44NtazCB3pa-07S-uK769gJ2ZcS`
- **Webhook** (manuell): `GET http://127.0.0.1:5678/webhook/pickliste-verarbeiten?file_id=<ID>`

**Ablauf:**
1. Datei-ID ermitteln (aus welchem Trigger)
2. `Fehler_und_Hinweise` lesen → **Idempotenz-Check**: Datei schon verarbeitet? → Skip
3. PDF von Google Drive herunterladen
4. Claude (Haiku) parst PDF → `[{name: "Nayax-Name", pick: Zahl}]`
5. Für jedes Produkt:
   - Suche aktive Slots via `nayax_product_name`
   - `effective_qty = min(pick, sum(remaining_qty aller aktiven Chargen))`
   - Greedy Slot-Befüllung: erst Slot 1 bis Kapazität, dann Slot 2 mit Rest
   - Update `current_machine_qty` in Produkte
6. Offene Warnungen (`EMPTY_BATCH`, `LOW_STOCK`, etc.) für betroffene Produkte → `resolved=TRUE`
7. Audit-Einträge (`PICKLISTE_REFILL` pro Produkt + `PICKLISTE_VERARBEITET` als Idempotenz-Marker)
8. Datei von `pickliste unbearbeitet` → `pickliste archiviert` verschieben

**Idempotenz-Schutz (Doppelverarbeitung):**
Vor der Verarbeitung prüft Code-Idempotenz ob bereits ein `PICKLISTE_VERARBEITET`-Eintrag
in `Fehler_und_Hinweise` mit `DATEI:<filename>` existiert.
Falls ja → Workflow antwortet mit `{skip: true}` und macht nichts.

**Manuelle zweite Absicherung:**
Test-Pickliste nach erfolgreichem Test manuell aus „pickliste unbearbeitet" entfernen
(solange noch nicht aufgefüllt wurde), damit der Drive-Trigger nicht feuert.

**Patch-Skript:** `guv_check_tmp/patch_wf9_pickliste.js`
**Lokales JSON:** `WF9 - Pickliste verarbeiten.json`

blieben EMPTY_BATCH / LOW_BATCH / INSUFFICIENT_BATCH_STOCK Einträge in

- `Extract - Auto-Resolve Hinweise` (Code)
- `Google Sheets - Hinweise auflösen` (GSheets update, matched auf `created_at`)

**Logik in `Code - MHD und Lagercharge pruefen` (am Ende, nach Alert-Generierung):**

```
Für jeden ungelösten Hint in Fehler_und_Hinweise:
  EMPTY_BATCH             → auto-resolve wenn totalRemaining > 0
  INSUFFICIENT_BATCH_STOCK → auto-resolve wenn totalRemaining > 0
  LOW_BATCH               → auto-resolve wenn totalRemaining > lowBatchThreshold (5)
  LOW_STOCK               → auto-resolve wenn current_machine_qty >= min_stock
```

Marker: `// PATCH AUTO_RESOLVE_WF5`

**WF5 läuft täglich um 07:00** — also wird jede Verbesserung innerhalb von
24 Stunden automatisch erkannt und aufgelöst.

## Konzept: Auto-Refill-Inferenz im WF3 (Modell A)

User-Anforderung: Auffüll-Vorgang am Automaten soll NICHT manuell ins Sheet
nachgepflegt werden müssen. Slot-Updates dürfen Lager NICHT beeinflussen.

Modell-Semantik (unveränderlich):
- `Lagercharge.remaining_qty` = TOTAL physisch (Slot + Backstock)
- `Produkte.current_machine_qty` = nur Slot (Subset)
- Slot-Updates → Lager bleibt unverändert (matchingColumns verhindert das)
- Verkauf → beide reduziert

Neu in WF3 `Code - FIFO berechnen` (Funktion `queueProductQtyDeduction`):
```
Bei jedem Sale:
  Wenn current_machine_qty <= 0 UND Lager.remaining > 0:
    → infer: User hat Slot aufgefüllt
    → setze current_machine_qty = MIN(machine_capacity, lager_remaining)
    → DANN normaler Abzug -1
    → Audit-Warning AUTO_REFILL_SLOT (severity=info)

  Wenn current_machine_qty <= 0 UND Lager.remaining = 0:
    → INSUFFICIENT_BATCH_STOCK (Lager wirklich leer)
    → Email-Alert: "Bitte neue Lagercharge ergänzen"
```
